в—Ҹ Good script structure, but there are several issues that will cause failures
  or unexpected behaviour. Let me walk through them and provide a corrected
  version.

  Issues Found

  1. Az context not inherited by parallel runspaces вҖ” Set-AzContext only affects
   the current runspace. Each -Parallel scriptblock runs in its own runspace
  with no Az context. This will fail silently or throw auth errors.
  2. -ScriptPath is local, not remote вҖ” Invoke-AzVMRunCommand -ScriptPath reads
  the file from the machine running the script and sends its contents to the VM.
   That's correct behaviour, but worth confirming
  C:\temp\IME_Device_CSATCloudScan.ps1 exists on your admin workstation (not the
   target VMs).
  3. No VM power state check вҖ” Deallocated VMs will fail and waste time waiting
  for timeouts.
  4. No OS filter вҖ” If any Linux VMs exist in those subscriptions,
  RunPowerShellScript will fail against them.
  5. Incomplete exclusions вҖ” You mentioned excluding AAD Connect and Exchange
  2019 management VMs by name, but the script only excludes by resource group.
  6. No timeout вҖ” Invoke-AzVMRunCommand can hang for 20+ minutes on unresponsive
   VMs. No built-in timeout parameter exists, so you need a wrapper.
  7. Output path вҖ” C:\temp\ assumes you're running from Windows PowerShell, not
  WSL. Since you have PS7 in WSL, decide where this runs.

вқҜ so how is the script working - is the csat script being copied to the vm and  
run from there, or is it all being done locally via my machine?                 

в—Ҹ Good question. Here's exactly what happens:

  1. Your workstation reads C:\temp\IME_Device_CSATCloudScan.ps1 into memory
  (line 105: Get-Content $ScriptPath -Raw)
  2. Invoke-AzVMRunCommand sends the script content to the Azure VM Agent via
  the Azure fabric (not over the network directly)
  3. The VM Agent receives the script, writes it to a temp location on the VM,
  and executes it under SYSTEM context on the remote VM
  4. StdOut/StdErr are captured by the VM Agent and returned back to your
  workstation via the Azure fabric

  So the CSAT script executes on each VM remotely вҖ” your workstation is just the
   orchestrator. The script never runs locally on your machine.

  Important implication: Since Run Command executes under SYSTEM context on the
  VM, this is the same context that failed with the GPO approach. If the CSAT
  script's AzCopy download from aka.ms or blob upload fails under SYSTEM due to
  proxy/network issues, you'll see the same failure вҖ” but this time you'll get
  the error captured in StdErr in the CSV, which gives you visibility you didn't
   have before.

  Check the batch 1 results CSV when it finishes вҖ” if you see errors related to
  AzCopy or network downloads, that confirms the SYSTEM context is the root
  cause and we'd need to address proxy/firewall rules for SYSTEM on those VMs.

вқҜ what extension does the "Run Command" belong to in Azure VM's?                

в—Ҹ There are two versions:

  - Run Command v1 (what this script uses via Invoke-AzVMRunCommand) вҖ” no
  separate extension needed. It works through the Azure VM Guest Agent
  (WindowsAzureGuestAgent) that's already installed on every Azure VM. It's a
  built-in capability of the agent.
  - Run Command v2 (the newer az vm run-command create / Set-AzVMRunCommand) вҖ”
  uses the RunCommandHandler extension
  (Microsoft.CPlat.Core.RunCommandHandlerWindows). This version supports
  longer-running scripts, async execution, and output to blob storage.

  Since we're using Invoke-AzVMRunCommand (v1), the only dependency is that the
  VM Guest Agent is running and healthy on each target VM. If a VM has a stale
  or unresponsive agent, the command will hang until timeout.

  If you're asking because of the earlier ASR extension work or want to verify
  the agent is healthy across the fleet, you can check with:
